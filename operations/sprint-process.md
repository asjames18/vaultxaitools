# Sprint Process — Melanated In Tech

## Overview

MIT runs **2-week sprints** on a rolling cadence. Sprints begin on Monday and end on the Friday of the second week. AI agents participate as first-class contributors alongside human team members, with tasks assigned, tracked, and reviewed the same way regardless of contributor type.

---

## Sprint Cadence

| Day | Event |
|-----|-------|
| Monday (Week 1) | Sprint Planning |
| Wednesday (both weeks) | Async Standup check-in |
| Friday (Week 2) | Sprint Review |
| Friday (Week 2, after Review) | Retrospective |

---

## Ceremonies

### Sprint Planning (Monday, Week 1 — 90 min)
- Product Owner presents prioritized backlog
- Team pulls items into the sprint using MoSCoW method (see below)
- AI agents flag tasks they can execute autonomously vs. tasks requiring human approval gates
- Each item gets an assignee (human or agent), acceptance criteria, and a Definition of Done check
- Output: locked sprint board in Linear

### Async Standup (Wednesdays)
- Each contributor (human or agent) posts a structured update:
  - **Done:** completed since last standup
  - **Doing:** in-progress work
  - **Blocked:** anything preventing progress
- AI agents post standup summaries via automated workflow to the team Slack channel
- Blockers are triaged within 4 hours

### Sprint Review (Friday, Week 2 — 60 min)
- Demo all completed work against acceptance criteria
- Product Owner formally accepts or rejects each item
- Rejected items return to backlog with documented reasons
- AI agent outputs (code, content, configs) reviewed by a human before acceptance

### Retrospective (Friday, Week 2 — 45 min)
- Format: What worked / What didn't / What we change
- AI agents contribute a written retro summary based on sprint metrics (velocity, blockers, cycle time)
- Action items assigned to owners with a due date before the next sprint starts

---

## Definition of Done

A backlog item is Done when ALL of the following are true:

- [ ] Acceptance criteria fully met
- [ ] Code reviewed (human review required for AI-authored code touching payments, auth, or schema)
- [ ] Tests written and passing (unit + integration where applicable)
- [ ] Deployed to staging and smoke-tested
- [ ] Changelog entry written
- [ ] Documentation updated if the feature is user-facing
- [ ] Product Owner has accepted the item

---

## Prioritization — MoSCoW Method

Each sprint planning session, backlog items are categorized:

- **Must Have** — Sprint fails without this. Blocks a release or a core user flow.
- **Should Have** — High value, ships this sprint if capacity allows.
- **Could Have** — Nice to have. Pulled in only after Must and Should are covered.
- **Won't Have (this sprint)** — Explicitly parked. Documented so it isn't re-debated mid-sprint.

AI agents may propose priority shifts based on dependency analysis, but priority changes after planning are locked unless a P0/P1 incident forces replanning.

---

## Velocity Tracking for Mixed Human + AI Teams

- Velocity is measured in **story points completed per sprint**
- Human and AI contributions are tracked separately in Linear using contributor tags
- A sprint health report is auto-generated by the ops agent at the end of each sprint:
  - Points planned vs. completed
  - AI agent task completion rate
  - Spillover percentage
- Rolling 4-sprint average is used for planning capacity, not a single sprint's velocity
- AI agent velocity is expected to be higher on well-scoped technical tasks; human capacity is prioritized for strategic decisions, design, and stakeholder-facing work
